# Гномобій — план великого рефакторингу та контенту

> Детальний план для виконання в Claude Code Web. Стан репозиторію на момент
> складання: гілка `main`, коміт `d00f46a`, робоче дерево чисте, версія 4.1.1.
> Усі тести зелені: `--test` та `--test-story` (25/25).

## 0. Як працювати над цим планом (правила для виконавця)

1. **Тест-драйв.** Проєкт має headless-харнес. Після **кожного** етапу ганяй:
   ```
   godot --headless --path . --import
   godot --headless --path . -- --test          # PvE smoke
   godot --headless --path . -- --test-story     # повний сюжетний цикл
   ```
   Плюс МП-смоук (`--mphost` + `--mpjoin` у двох терміналах) після змін у `net.gd`,
   `game.gd`, `player.gd`, `gnome.gd`. Жоден етап не вважається завершеним, поки не
   `PASS=true` на всьому та немає `SCRIPT ERROR`.
2. **Малими PR.** Кожен під-етап = окремий коміт із доробленим повідомленням. Ніколи
   не змішувати рефакторинг (зміна форми без зміни поведінки) з новою поведінкою в
   одному коміті — інакше регресії неможливо локалізувати.
3. **Спершу рефакторинг файлів, які потім чіпатимуть інші воркстріми** (`game.gd`,
   `world_gen.gd`, `hud.gd`), щоб контент і механіки лягали вже в чисту структуру.
4. **Порядок рядка при `Set-Content`/збереженні — LF.** У репо `.gitattributes`
   нормалізує CRLF; зберігай файли з LF, щоб не було шумних діфів.
5. **Не чіпати мережевий контракт наосліп.** `net.gd` — авторитетний сервер; будь-яка
   зміна RPC/структур стану має лишатися сумісною або піднімати `GAME_VERSION`.
6. **Мова коду:** коментарі — російською (як у всьому проєкті), UI-рядки — через
   `tr()` з російським ключем (див. воркстрім B).

`medieval` (4 будівлі); зброя героя (8) та скелети — усі на місці.

## Воркстрім A — Рефакторинг великих файлів

**Мета:** SRP-модулі, ціль ≤300 рядків/файл. Реалістичний виняток: вузли з великою
кількістю пов'язаного стану/RPC (`net.gd` RPC-поверхня, ядро `gnome.gd`) можуть лишитись
до ~400 — це нормально, не роздувай штучно. Кожен спліт — **чистий рефакторинг**:
поведінка ідентична, тести зелені до і після.

**Патерн спліту в GDScript:**
- `static`-утиліти (`world_gen.gd`) → окремі класи зі `static func`, оновити виклики.
- Логіку вузла (`game.gd`, `player.gd`, `gnome.gd`, `hud.gd`) → винести когезивні
  системи в окремі скрипти-компоненти (`RefCounted` або дочірній `Node`), що тримають
  посилання на власника й оперують його станом. Game володіє компонентами як полями.
- Не ламати автолоади (їх 8; вони — singleton-класи, розбивати лише винесенням
  `static`-хелперів у сусідні файли).

### A1. `main.gd` → винести тест-харнес (найлегша й найбільша перемога)
- Створити `scripts/test_harness.gd` (клас `TestHarness`, `RefCounted` або `Node`).
- Перенести весь код `--test`/`--test-story`/`--screenshot`/`--mphost`/`--mpjoin` з
  `main._process` та супутні поля (`_test_mode`, `_test_timer`, `_shot_stage`, …).
- `main.gd` лише детектить cmdline-прапорці й делегує в `TestHarness.tick(delta)`.
- **Очікування:** `main.gd` втрачає ~600 рядків; усі тестові режими працюють як раніше
  (це і є перевірка — харнес тестує сам себе).

### A2. `main.gd` → меню й налаштування
- `scripts/ui/menu_builder.gd` — побудова екранів (play / multiplayer / achievements).
- `scripts/ui/settings_ui.gd` — вкладки налаштувань (спільні для меню й паузи, зараз
  ~786-1040).

  `scripts/chapter_flow.gd` для переходів між главами, ~1075-1300).
- **Ціль:** `main.gd` ≤ ~300.

### A3. `game.gd` (3259) → сервіси, якими володіє Game
Винести когезивні системи (кожна ~150-350 рядків), Game тримає їх як поля й делегує:

## Воркстрім B — Локалізація (магазин та інші прогалини)

Архітектура правильна; це **5 точкових багів** + захист від регресій. Ключі додавати в
`locale/translations.csv` (колонки keys/ru/uk/en), після чого `--import` перекомпілює
`.translation`.

### B1 (головний — «немає перекладів у магазині»)
`Items.def_name()` (`items.gd:83-89`) знає лише `WEAPONS`/`TRINKETS`, для розхідників
повертає сирий id → магазин показує `potion_hp`, `bomb`, `potion_rage`, `potion_speed`
у **всіх** мовах (і в російській теж). Коректні назви є в `game.gd:27-33 ITEM_DEFS` і в
CSV, але `def_name` до них не доходить.
- **Фікс:** навчити `def_name` (або шар магазину/інвентарю) резолвити id розхідника →
  назву. Оскільки `items.gd` не має доступу до `Game.ITEM_DEFS`, найчистіше — додати в
  `items.gd` мапу id→російська-назва для розхідників (ті самі ключі, що вже в CSV:
  «Зілля здоров'я» тощо) або передавати назву ззовні. Показ іде через `tr()`, тож
  переклади вже є.
- Перевірити і колонку **продажу** (`hud.gd:803`) — той самий шлях.

### B2 — PvP-банер перемоги (`game.gd:542`)
`"%s ПОБЕЖДАЕТ!" % name` підставляє ім'я **до** `tr()` (`game.gd:1196`), тож рядок
ніколи не збігається з ключем → лишається російським в uk/en. Ключ `"%s ПОБЕЖДАЕТ!"` у
CSV є.
- **Фікс:** передавати шаблон, а підставляти після перекладу: на клієнті
  `tr("%s ПОБЕЖДАЕТ!") % name`.

### B3 — мишачі підписи `key_name()` (`main.gd:112-115`)
`"ЛКМ"/"ПКМ"/"СКМ"/"Мышь %d"` без `tr()` і без ключів → російські в uk/en (і на кнопках
кібіндів, і в підказках типу `main.gd:912`).
- **Фікс:** обгорнути в `tr()`, додати 4 ключі в CSV з uk/en.

### B4 — коди хотбара (`hud.gd:979`)
`s.short.text = def.get("short", "?")` бере `"ЗД"/"ЯР"/"СК"/"БМ"/"ПИР"` з
`game.gd:28-32` без `tr()` і без ключів.
- **Фікс:** обгорнути в `tr()`, додати ключі (або локалізувати короткі коди інакше).

### B5 — запасне ім'я гравця `"Друг"` (`game.gd:619/640/645`)
Не обгорнуте, немає ключа.
- **Фікс:** `tr("Друг")` + ключ.

### B6 — захист від регресій (guard)
Додати в тест-харнес (воркстрім A1) перевірку локалізації: пройтись відомими
динамічними шляхами (усі id розхідників через `def_name`, усі `short`-коди, усі
`RARITY_NAMES`, назви класів зброї/тринькетів/афіксів, назви скілів/ачивок/квестів) і
впасти `PASS=false`, якщо `tr(x) == x` там, де x — кирилиця/сирий id. Це ловить майбутні
пропуски автоматично. Опційно: маленький standalone-скрипт, що звіряє всі
`tr("literal")` у `scripts/*.gd` з ключами CSV.

**Acceptance воркстріму B:** у трьох мовах магазин, інвентар, PvP-банер, підказки
керування й хотбар — без сирих id та без чужомовних вкраплень; guard зелений.

## Воркстрім C — Повноцінний рівень-дизайн

**Директива користувача:** зараз «пустувато, але велике» → зробити **насиченим,
інтерактивним, але не великим**; різні локації, підземелля, не лише поляни.

### C0. Принципи
- **Менше площі, більше вмісту.** Зменшити `OVERWORLD_RADIUS` 120 → **~80** (диск ~160
  в поперек). Стягнути 5 областей ближче; прибрати мертві порожнини між ними.
- **Композиція, не розсип.** Локації будувати модульно (як генератор підземель), а не
  розкиданими пропами: у кожної — впізнаваний силует і призначення.
- **Щільність = інтерактив.** Наповнювати не деревами, а взаємодіями: схованки,
  міні-сутички, руйнівні об'єкти, події оточення.

### C1. Переробка оверворлда (`wg_overworld.gd`)
- **Стягнута мапа-подорож:** камп → село → роздоріжжя (поле бою / гай) → підхід → крипта,
  але на радіусі ~80 з коротшими перегонами. Дорога лишається хребтом (плитки
  `halloween/path_*`), але з **відгалуженнями-тупичками до схованок** (скриня/лор/кеш).
- **Насиченість областей:** кожна область отримує «ядро» з 3-5 згрупованих об'єктів
  (напр. село: криниця + ринкові ятки + багаття + NPC довкола площі; поле бою: зламані
  паркани, встромлені мечі `sword_shield_broken`, вогнища, тіла), а не 1 POI + порожнеча.
- **Нові типи локацій (різні «локи»)** — зібрати з наявних ассетів:
  1. **Занедбане село/руїни** — `medieval` будівлі + `dungeon/wall_broken`,
     `rubble_large`, `column`; частина будинків «зруйнована».
  2. **Цвинтар** — `halloween` grave_A/B, gravemarker, skull, dead trees, lanterns
     (вже є як night-біом — виокремити в самостійну область із криптами-схованками).
  3. **Застава/блокпост** — паркани + ворота + вежа зі `column`/`wall` + смолоскипи;
     міні-сутичка на вході.
  4. **Табір ворога** — кільце хатин-спавнерів довкола вогнища з міні-босом (еліта).
  5. **Переправа/міст** — вузьке горло (вибір, де тримати оборону), обабіч — вода-декор
     (площина шейдера) або яр.
  - Області вибираються по біому/главі, щоб глави відрізнялися компоновкою, а не лише
    палітрою.
- **Більше інтерактиву в світі:**
  - Руйнівні об'єкти: бочки/діжки (`barrel_large`, `keg`) — вибухові/з лутом при ударі.
  - Приховані кеші: `chest`/`coin_stack` за деревами/у руїнах, підказка через блиск.
  - Події оточення: засідка (гноми вилазять при вході в горло), сигнальні вогнища
    (вже є в сайд-квесті — розширити), пастки на стежках.
- **Прибрати наповнювач:** різко скоротити лісові пояси (240 → ~120 дерев), натомість
  додати згруповані «гаї» й кущі лише де це формує простір/укриття.

### C2. Різноманіття підземель (`world_dungeon.gd` → розбити на модуль)
- **Кілька тем/рецептів** замість одного грид-генератора: `crypt` (поточний),
  `cave`/`mine` (нерегулярні печери, `trunk_*`, `rubble`, менше прямих стін),
  `catacombs` (щільні коридори, більше розгалужень). Тема — по біому/главі.
- **Багатша структура:** розгалуження й **петлі** (не лише ланцюг L-коридорів),
  **секретні кімнати** (прихована стіна, що відкривається інтеракцією), **замкнені двері
  + ключ** (проста прогресія всередині рівня), **міні-бос** окрім фінального.
- **Небезпеки оточення** понад шипи: падаючі решітки, вогняні жарівні (`torch`+урон),
  провали в підлозі, рухомі пастки — усі детерміновані від сіда й серверні.
- **Кімната-нагорода** з вибором лута (взяв одне — інше зникає) для реіграбельності.
- **Acceptance:** підземелля будується для 3 тем без помилок, навмеш печеться,
  вхід/вихід (крипта→підземелля→оверворлд) працює як раніше; `--test-story` (який
  проходить підземелля) зелений.

### C3. Тест-покриття рівнів
Розширити перевірки в тест-харнесі (як уже зроблено для перекриття колайдерів):
- 0 перекриттів колайдерів великих об'єктів (вже є — зберегти).
- Зв'язність: від спавну гравця досяжні (по навмешу) усі області та вхід у підземелля.
- Щільність: у кожній області ≥ N інтерактивних об'єктів; частка порожнього простору
  нижча за поріг.
- Ворота відкриті (стулки сховані), фонарі збоку від дороги (продовження 4.1.1-фіксів).

## Воркстрім D — Нові механіки + інтерактивний туторіал

### D1. Нові ігрові механіки (узгоджений, реалістичний набір)
Обрати підмножину, що пасує грі; кожна — окремий PR із тестом:
1. **Стаміна** (заповнює явну прогалину): біг/перекид/важка атака витрачають, реген
   у спокої; порожня стаміна блокує спринт/додж. Смуга в HUD. **Тюнити м'яко**, щоб не
   зламати відчуття; за потреби — вимикач у налаштуваннях складності.
2. **Парі (timed block):** блок у вікні ~0.2 с перед ударом → стаган ворога + вікно на
   фінішер; винагороджує майстерність, спирається на наявний блок.
3. **Заряджена важка атака** (утримання ЛКМ): більший урон/пробиття блоку, витрата
   стаміни.
4. **Статус-ефекти** від афіксів/зброї: підпал (dot), сповільнення, кровотеча —
   серверні, з іконками. Дає сенс новим афіксам.
5. **Вибухові/руйнівні об'єкти** (спільне з C1): бочки, жарівні.
6. **Бестіарій/кодекс** (пасивна прогресія знань): перше вбивство типу → запис; дешево
   й додає цілей.
7. (Опційно) **Крафт/готування біля багаття:** з луту робити зілля — зв'язує економіку
   й POI-багаття.
- Для кожної: серверна авторитетність (урон/кулдауни рахує сервер), реплікація через
  наявні канали, оновлення `--test`/`--test-story` з новими ассертами.

### D2. Інтерактивний туторіал для новачків
**Хуки вже є** (не треба будувати з нуля): FSM квестів (`q_main` як шаблон),
`hud.set_quest_lines()` (список цілей), `hud.set_hint()` (контекстна підказка),
`hud.show_dialog()` (typewriter-репліки), `hud.banner()`/`center_msg()`, чок-поінти дій
(`player.gd` interact-диспетч ~282, `game.use_item_slot` ~2999, `try_attack`/`try_dodge`).

**Флаг «грав уперше»:** додати `meta/tutorial_done` у `savegame.gd` (файл
`user://slots.cfg`, секція `meta` — там уже живе `active_slot`). Показувати туторіал,
коли починається нова кампанія і флаг не стоїть; ставити флаг по завершенні.

**Структура (паралельний FSM `q_tutorial`, за зразком `q_main`):**
- Виділена безпечна ділянка (розширити табір глави 1) або скриптована інтро-послідовність
  у сейф-зоні — без реального ризику смерті.
- Гейтовані кроки «зроби дію → крок зараховано → наступна підказка»:
  1. Рух (WASD) + камера (миша).
  2. Спринт (і стаміна, якщо додано).
  3. Атака й **комбо** (3 удари) по манекену/слабкому гному.
  4. Блок і **перекид** (i-frames) від телеграфованого удару.
  5. Взаємодія: відкрити скриню (E), поговорити з NPC.
  6. Інвентар (I): вдягнути зброю/тринькет.
  7. Аркуш персонажа (C): вкласти очко стату, вузол скіл-дерева.
  8. Хотбар (1-5): застосувати зілля.
  9. (Ко-оп) підняти збитого напарника (E) — або пояснення, якщо соло.
  10. Слідування за ціллю квесту (стрілка/маркер — див. нижче).
- **Візуальний вейпоінт:** зараз немає world-space маркера цілі (лише `!`/`?` над NPC і
  текст-трекер). Додати простий екранний індикатор напряму до поточної цілі (стрілка на
  краю екрана або 3D-маркер) — корисно і для туторіалу, і для гри загалом.
- **Пропуск:** кнопка «Пропустити туторіал» (ставить флаг), щоб не дратувати досвідчених.
- **Локалізація:** усі репліки/підказки туторіалу — через `tr()` з новими ключами в CSV
  (усі три мови).

**Acceptance воркстріму D:** новий `--test-tutorial` режим у харнесі проходить усі кроки
скриптовано (симулює дії, перевіряє просування `q_tutorial` і встановлення флага);
існуючі тести зелені; туторіал з'являється лише на першому запуску та за кнопкою в меню.

## 3. Верифікація (загальна стратегія)

- **Регресії форми (рефакторинг):** тести до/після мають давати ідентичні `[TEST]`-рядки
  (окрім навмисних нових ассертів). Порівнювати вивід.
- **Кілька сідів:** для рівень-дизайну ганяти `--test-story` на 5+ сідах (як у 4.1.1),
  щоб ловити рідкісні збої розміщення/зв'язності.
- **МП:** будь-яка зміна серверної логіки/RPC → `--mphost`+`--mpjoin`, host бачить
  обох, join без `SCRIPT ERROR` і без «версії не збігаються».
- **Візуальне (обмеження):** headless не рендерить; орієнтацію пропів, читабельність
  туторіалу, відчуття стаміни підтверджує людина. План це чесно фіксує — лишати
  контрольні точки для ручного огляду.

## 4. Ризики й застереження

- **300-рядковий поріг** недосяжний догматично для вузлів з RPC/тісним станом
  (`net.gd`, ядро `gnome.gd`). План ставить ~300 ціллю з обґрунтованими винятками до
  ~450; якщо потрібен жорсткий поріг усюди — це більше файлів, більший churn і вищий
  ризик регресій (рішення за власником).
- **Стаміна** змінює відчуття бою — вводити з тюнінгом і, можливо, вимикачем.
- **Обсяг величезний.** Це тижні роботи; виконувати мілстоунами, кожен — самодостатній
  і придатний до релізу.
- **Ассет-обмеження:** різноманіття будівель тонке (4 medieval, лише «green»). Багаті
  «локи» доведеться складати переважно з `dungeon` + `halloween` паків; за потреби —
  додати ще один CC0-пак оточення (окреме рішення).
